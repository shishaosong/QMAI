## 20260617-修复 AI 会话选 model 后本地 CLI 抢占调用的 bug

### 🎯 问题
用户反馈：AI 会话选择 `deepseek-v3`（来自 custom-* 卡片）后，调用失败，错误信息显示：
```
API Error: 503 No available channel for model claude-opus-4-6 under group default
(distributor) (request id: ...). This is a server-side issue, usually temporary
— try again in a moment. If it persists, check your inference gateway
(127.0.0.1:15721).
```

注意：
- **错误模型名是 `claude-opus-4-6`（不是用户选的 `deepseek-v3`）**
- **错误来自本地 gateway `127.0.0.1:15721`**

### 🔍 根因（user 猜测完全正确）
本地 Claude Code CLI / Codex CLI 在 `provider === "claude-code"` / `"codex-cli"` 时优先级太高：

1. `chat-panel.tsx` 之前只覆盖 model 字段，**provider 保持 activePresetId 的配置**：
   ```ts
   const effectiveChatLlmConfig = aiChatModel.trim()
     ? { ...llmConfig, model: aiChatModel.trim() }  // ← provider 不变
     : llmConfig
   ```
2. 如果用户的 `activePresetId = "claude-code"`（或 `"codex-cli"`），`llmConfig.provider` 自动同步为本地 CLI 类型
3. AI 会话选 `deepseek-v3`（来自 custom-ali 卡片）时，`provider` 仍为 `"claude-code"`
4. `llm-client.ts` 看到 `provider === "claude-code"`，**走 `streamViaClaudeCodeCli` 启动本地 CLI 子进程**
5. **本地 CLI 忽略 `model` 参数**，使用本地默认模型（`claude-opus-4-6`）
6. 本地 `.claude/settings.json` 配置了 `ANTHROPIC_BASE_URL=http://127.0.0.1:15721`（本地 inference gateway）
7. CLI 用 `claude-opus-4-6` 调本地 gateway → 503 错误（无可用 channel）

**本质：** user 选 model 时 `provider` 仍为本地 CLI，**本地 CLI 强制使用默认模型，覆盖 user 选择**。

### 🔧 修复

**`chat-panel.tsx`（handleSend 内联计算）：**
- 新增 `import { resolveConfig } from "@/components/settings/preset-resolver"`
- 新增 `import { LLM_PRESETS } from "@/components/settings/llm-presets"`
- 新增 `const providerConfigs = useWikiStore((s) => s.providerConfigs)`
- 重写 `effectiveChatLlmConfig` 计算逻辑：
  1. 通过 `aiChatModel` 查找它属于哪个 `providerConfigs` 卡片
  2. 找到 `LLM_PRESETS` 中对应的 preset（custom-* 用 `"custom"` 模板 fallback）
  3. 调用 `resolveConfig(template, override, llmConfig)` 重新计算完整 LlmConfig
  4. 用 `aiChatModel` 覆盖 `model` 字段
  5. 如果找不到匹配（fallback），回退到只覆盖 `model`（旧行为）
- handleSend 依赖列表加 `providerConfigs`

**修复后：** `effectiveChatLlmConfig.provider` 从 `"claude-code"` → `"custom"`，走 HTTP API 调用 deepseek 服务，不再走本地 CLI。

## 20260617-修复 updateCard 字段覆盖导致 savedModels 丢失的 bug

### 🎯 问题
用户反馈：在某个版本保存好自定义卡片模型后，**升级到新版本重新打开应用，所有模型选择记录都会消失**。

### 🔍 根因
`updateCard` 函数（line 80-101）显式列出每个字段赋值：

```ts
const updatedConfig: ProviderOverride = {
  ...providerConfigs[id],
  label: updates.label,          // ← 当 updates.label 是 undefined 时，
  apiMode: updates.apiMode,      //   会把 store.label 覆盖为 undefined！
  baseUrl: updates.baseUrl,
  apiKey: updates.apiKey,
  model: updates.model,
  maxContextSize: updates.maxContextSize,
  reasoning: updates.reasoning,
  savedModels: updates.savedModels,  // ← 任何非 savedModels 的更新都会清空这里
}
```

**问题链路：**
1. 用户保存了模型：`onUpdate({ savedModels: [...] })` → store 正确写入 savedModels
2. 用户修改其他字段（如 baseUrl）：`onUpdate({ baseUrl: "..." })`
3. `updateCard` 内部 `savedModels: updates.savedModels` → `undefined`
4. `setProviderConfigs` + `persistConfigs` 把 undefined 写入 store **和磁盘**
5. 用户重新打开应用 → cards state 从 store 读取 → `config.savedModels || []` → **空数组**
6. 模型"消失"

**setCards（line 81）行为正确**（`{...c, ...updates}` 保留未指定字段），但 **updateConfig 行为错误**（显式列出导致覆盖）。

### 🔧 修复
改用 `??` 回退到 store 中已有值：

```ts
const prev = providerConfigs[id] ?? {}
const updatedConfig: ProviderOverride = {
  ...prev,
  label: updates.label ?? prev.label,
  apiMode: updates.apiMode ?? prev.apiMode,
  baseUrl: updates.baseUrl ?? prev.baseUrl,
  apiKey: updates.apiKey ?? prev.apiKey,
  model: updates.model ?? prev.model,
  maxContextSize: updates.maxContextSize ?? prev.maxContextSize,
  reasoning: updates.reasoning ?? prev.reasoning,
  savedModels: updates.savedModels ?? prev.savedModels,
}
```

**修复效果：**
- 当 `updates.X` 是 `undefined` 时，保留 store 中原值
- 当 `updates.X` 是有效值（包括 `""` 和 `[]`），正常覆盖
- 当 `providerConfigs[id]` 不存在（新增卡片场景），用空对象回退——但 `addCard` 不走 `updateCard`，所以这个回退主要是防御性的

### 📂 改动文件
- `src/components/settings/sections/custom-provider-cards.tsx`
  - `updateCard` 函数：所有显式字段改为 `updates.X ?? prev.X` 形式
  - 新增 `const prev = providerConfigs[id] ?? {}` 防御性回退
