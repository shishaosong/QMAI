## 20260616-大纲强制读取优化

### ✨ 功能改动
- **移除大纲长度限制**：`readOutlineContent()` 函数不再限制大纲在12000字符内，现在读取完整大纲内容（之前限制每个文件2500字符，最多12个文件，总计12000字符）
- **大纲独立置顶传递**：大纲不再混在18个context section中，而是独立提取并在每个生成阶段的提示词最前面呈现，带有强制遵守的明确声明
- **大纲永不被裁剪**：即使启用token预算管理（32000字符），大纲也不会参与裁剪逻辑，确保AI始终能看到完整大纲
- **所有阶段强制包含大纲**：任务书、初稿、扩写、审稿、润色等所有阶段都强制包含完整大纲，确保AI在每个环节都严格遵守大纲

### 🎯 问题解决
- **问题描述**：当生成内容超过20章左右时，AI生成的章节会偏离大纲内容
- **根本原因**：
  - 大纲被限制在12000字符以内，长大纲被截断
  - 大纲混在18个context section中排第4位，在长上下文中被稀释
  - 第20章+时累积内容膨胀，大纲占比从50%降到35%以下
  - 大纲可能被token预算裁剪
- **解决方案**：
  1. 移除所有大纲长度限制，读取完整内容
  2. 大纲独立提取并在提示词最前面强制呈现
  3. 大纲排除在token预算裁剪之外
  4. 在5个提示词构建函数中都增加outline参数并置顶

### 📂 改动文件
- **修改（6 个）**：
  - `src/lib/novel/context-engine.ts` — 修改`readOutlineContent()`移除长度限制；修改`contextPackToPrompt()`增加`excludeOutline`选项

## 20260615-22:50 - 导入预检 race condition 与 state 残留修复（fix/precheck-race-and-state-residue）

### 🐛 Bug 修复
- **导入任何小说文件，打开都是以前的角色**

#### 根因 1：precheck work 异步完成时覆盖 `setExistingEntry`
- 位置：[book-analysis-input-dialog.tsx#L120-L156](file:///e:/QMAI/src/components/novel/book-analysis-input-dialog.tsx#L120-L156)
- 表现：用户选了一个大文件（5MB+，readFile 慢），预检 5s 超时返回 null；用户又选了一个新文件，新预检完成后，**旧 work 终于完成时调 `setExistingEntry(旧_entry)` 覆盖了 `setExistingEntry(新_entry)`**，UI 显示旧 entry
- 用户点"打开现有结果"按钮 → 看到旧角色
- 修复：用 `useRef<AbortController>` 跟踪预检，新预检开始时 abort 旧预检，work 在每个 `await` 前检查 `signal.aborted`，**已 abort 的 work 不会触发 setState**

#### 根因 2：dialog 重新打开时 state 残留
- 表现：用户选了文件、关闭 dialog（点 X / ESC / 遮罩）、重新打开 dialog，**看到上次残留的 filePath 和"已分析过"提示**
- 背景：base-ui Dialog 在点 X / ESC / 遮罩关闭时只调 `onOpenChange(false)`，不调 `handleCancel`，导致 state 残留
- 修复：
  - 加 `useEffect` 监听 `open` 变化，open: false → true 时重置所有 state
  - 包装 `onOpenChange` → `handleOpenChange`，关闭时统一 abort 预检 + 重置 state
  - `handleSubmit` / `handleOpenExisting` / `handleCancel` 改用 `handleOpenChange(false)` 统一收口

## 20260615-07:40 - 根因修复：识别失败 `(0, t.createHash) is not a function`

### 🐛 真实根因
`character-recognition-engine.ts` 第 1 行：
```ts
import { createHash } from "crypto"
```
`crypto` 是 **Node.js 内置模块**。Vite 把它打到 Tauri 渲染层后，浏览器里 `t.createHash` 是 `undefined`，调用即抛错。

> 这是个经典坑：vitest 用 jsdom + Node API 时**测试能过**，但浏览器运行时直接挂。

### 🔧 修复
把 `stableCharacterId` 改成 **纯 JS 双 32-bit FNV-1a 拼接**（无任何依赖）：
- 第一段 hash 用字符 charCode
- 第二段 hash 混入 charCode + 位置偏移降低冲突
- 输出 16 hex 字符 → `slice(0, 12)` 与原 SHA-256 截断后行为一致
- 跨 Node.js / 浏览器 / Tauri 都能跑
